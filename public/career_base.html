PROJECT 01

AIBOO — Edge AI 음성 비서 (Jetson Orin)

공공기관 노인 케어 음성 비서. 오픈소스 음성 비서 구조를 베이스로, Jetson 단일 장치 운영에 맞게 런타임을 재구성하고 공공기관 케어 도메인 기능을 확장 구현.

기간
2025.02 ~ 현재 (2026.05 납품 예정)
역할
1인 용역 — 백엔드, 대시보드, Jetson 런타임, ESP32 오디오 펌웨어 수정·튜닝, 케어센터 API 연동 담당
성과
Jetson Orin 런타임 메모리 약 750MB 절감, 단일 장치 ASR·LLM·TTS·센서·알림 통합
스택
Jetson Orin · aiohttp · Faster-Whisper · CTranslate2 · Silero-VAD · Piper TTS · MQTT · SQLite · PM2 · ESP-IDF

개요

시니어 사용자가 음성만으로 건강 점검, 일정 관리, 긴급 알림, 정서 지원을 받을 수 있는 온디바이스 음성 비서입니다. 외부 인터넷 의존을 낮춰야 하는 현장 조건에 맞춰 음성 인식, LLM, 음성 합성, 센서 이벤트 처리를 Jetson Orin 중심으로 통합했습니다.

문제

  • 기존 베이스 구조는 음성 인식·LLM·음성 합성·조율 컴포넌트가 Docker 4컨테이너로 분리되어 Jetson 단일 장치에서 메모리와 운영 부담이 컸습니다.
  • 공공기관 케어 도메인에는 일반 음성 비서 기능 외에 센서, 스케줄, 긴급 알림, 케어센터 API 연동이 필요했습니다.
  • Jetson 환경에서는 PyPI 기본 빌드가 맞지 않는 라이브러리가 있어 런타임 버전 조합을 직접 고정해야 했습니다.

핵심 기여

  • 메모리 약 750MB 절감. 기존 Docker 4컨테이너 구조를 단일 Python 프로세스로 통합하고, aiboo/mcp/homecare 핵심 서비스만 PM2로 분리 운영.
  • 단일 Jetson 시스템 통합. Faster-Whisper ASR, LLM provider, Piper TTS, MQTT 센서 이벤트, 스케줄, 긴급 알림, 케어센터 API 연동을 하나의 런타임 흐름으로 연결.
  • Jetson 런타임 환경 구축. PyTorch, CTranslate2, ONNX Runtime의 검증된 버전 조합을 환경 가이드에 고정하고 CTranslate2 CUDA 빌드를 직접 구성.
  • 운영 데이터 저장 안정화. 스케줄, MQTT, WebSocket 컴포넌트의 동시 쓰기 상황을 고려해 SQLite WAL 모드를 적용.
  • ESP32 오디오 스트리밍 수정·튜닝. ESP-IDF 기반 펌웨어에서 I2S Split 구성과 XVF3800 I2C 초기화 시퀀스를 반영해 마이크 입력/스피커 출력 흐름을 안정화.

성과/결과

  • 공공기관 현장 투입 예정 시스템의 소프트웨어 전 영역을 담당했고, 2026.05 납품 단계까지 개발 진행.
  • ASR·LLM·TTS·센서·스케줄·긴급 알림·케어센터 연동을 단일 Jetson Orin 장치에서 동작하도록 구성.
  • 후속 단계에서는 운영 데이터 기반 latency/resource 계측 체계를 추가할 예정.
PROJECT 02

Qwen LoRA 파인튜닝 + 로컬 LLM 배포 파이프라인

AIBOO 음성 비서의 로컬 LLM 후보를 만들기 위해 합성 데이터 생성, LoRA 학습, 모델 병합, GGUF 양자화, Ollama 등록까지 자동화.

기간
2025.12 ~ 현재
역할
합성 데이터셋 구축 · LoRA 학습 · GGUF 변환 · Ollama 서빙 자동화
성과
Qwen 계열 4개 모델 크기 비교 학습 및 로컬 서빙 파이프라인 구축
스택
Qwen · PEFT · TRL · llama.cpp · GGUF Q4_K_M · Ollama

개요

외부 LLM API에 의존하기 어려운 환경을 전제로, 로컬 상위 OSS LLM을 이용해 Self-Instruct 방식의 합성 데이터셋을 만들고 Qwen 계열 모델을 LoRA로 학습했습니다. logits distillation이 아니라 출력 텍스트 기반 SFT 데이터셋을 구성한 흐름입니다.

핵심 기여

  • Self-Instruct 합성 데이터셋 구축. 로컬 상위 OSS LLM을 이용해 다양한 상황·감정 톤의 instruction/output 데이터를 생성하고 중복을 필터링.
  • 음성 입력 최적화. 구두점 최소화, 구어체, 간투사, 불완전한 띄어쓰기 등 STT 결과에 가까운 입력 스타일을 데이터 생성 단계에서 강제.
  • 4개 Qwen 크기 비교 학습. Qwen2.5 0.5B/1.5B, Qwen3 1.7B/4B 계열을 LoRA로 비교해 품질과 Jetson 추론 비용의 균형점을 탐색.
  • 로컬 서빙 자동화. LoRA 어댑터 merge → HF 모델 → GGUF 변환 → Q4_K_M 양자화 → Ollama Modelfile 등록까지 자체 스크립트로 연결.

성과/결과

  • AIBOO LLM 모듈 후보를 외부 API 없이 반복 학습·변환·로컬 서빙할 수 있는 파이프라인 확보.
  • 정성 평가 기반으로 후보 모델을 선별했고, 후속 단계에서 자체 eval set과 LLM-as-judge 기반 정량 평가 체계를 도입 예정.
PROJECT 03

Multi-NPU 모델 포팅 (Kneron KL630 + Rockchip RK3588)

ONNX 이후 변환 단계에서 NPU 미지원 연산을 분리하고, 모델 본체는 NPU에 올린 뒤 후처리는 CPU SDK 코드로 구성한 객체 탐지 배포 파이프라인.

기간
2025.02 ~ 2025.09
역할
NPU 포팅 · ONNX 그래프 편집 · 후처리 분리 · 변환 파이프라인 구성
성과
Kneron KL630, Rockchip RK3588 NPU 2종 추론 파이프라인 검증
스택
Kneron Toolchain · RKNN Toolkit · ONNX · Netron · C · Docker

문제

YOLO 계열 객체 탐지 모델의 출력 후처리(NMS/detect 등)는 NPU에서 바로 지원되지 않는 경우가 많습니다. 전체 모델을 그대로 변환하기보다, NPU가 처리할 본체와 CPU가 처리할 후처리를 분리해야 했습니다.

핵심 기여

  • ONNX 그래프 수술. NMS/detect 등 NPU 미지원 후처리 노드를 ONNX 그래프에서 제거하고 모델 본체 출력이 유지되도록 절단·재연결.
  • Kneron KL630 변환. ONNX → BIE → NEF 파이프라인을 구성하고 ktc 양자화/컴파일 파라미터를 환경에 맞게 조정.
  • Rockchip RK3588 변환. RKNN Toolkit 기반 INT8 변환과 추론 검증 흐름 구성.
  • CPU 후처리 통합. SDK 레퍼런스 후처리(anchor decode, dequantization, sigmoid, NMS)를 임베디드 실행 환경에 맞게 통합.

성과/결과

  • NPU 2종에 대해 모델 본체 NPU 추론 + CPU 후처리 분리 구조를 구성하고 추론 동작을 검증.
  • ONNX 그래프 편집과 NPU toolchain 변환 사이의 실무 제약을 직접 다룬 경험 확보.

Fire/PPE Detection + RTSP 모니터링

산업 현장 안전 시나리오를 위한 YOLOv5s 6클래스 학습과 실시간 RTSP 모니터링 구현.

기간
2025.02 ~ 2025.09
역할
데이터셋 정합 · YOLOv5 학습 · RTSP 추론/대시보드 구현
성과
YOLOv5s 6클래스 mAP@0.5 84.7%, Multi-NPU 포팅 모델 소스 제공
스택
YOLOv5 · PyTorch · OpenCV · Streamlit · RTX 4090

핵심 기여

  • mAP@0.5 84.7% 달성. 화재, 연기, 안전모, 안전조끼, 사람, 머리 6클래스 YOLOv5s 모델 학습.

  • 희소 클래스 보강. 부족한 클래스는 복제와 변형을 적용해 학습 데이터 불균형을 완화.
  • RTSP 모니터링. OpenCV 기반 실시간 추론과 Streamlit 대시보드를 연결해 영상/탐지 로그를 시각화.

성과/결과

PROJECT 05

MCP 도구 릴레이 서버 (FastMCP 기반)

AIBOO에서 외부 도구 호출을 안정적으로 연결하기 위한 3계층 WebSocket 릴레이 서버.

기간
2026.03 ~ 현재
역할
릴레이 구조 설계 · WebSocket 라우팅 · 인증 · 테스트 자동화
성과
소비자 ↔ 서버 ↔ 파이프 ↔ stdio 도구 구조와 53개 테스트 케이스 구성
스택
FastAPI · FastMCP · WebSocket · pytest · Gemini · Kakao Local

핵심 기여

  • 3계층 WebSocket 릴레이. 소비자, 서버, 파이프, stdio 도구를 분리해 AIBOO와 MCP 도구를 연결.
  • JSON-RPC ID 변환. 여러 소비자의 요청 ID를 릴레이 내부 ID로 매핑하고 응답 시 원래 ID로 복구.
  • 연결 안정화. stdio ↔ WebSocket 파이프에 exponential backoff 자동 재연결을 적용.
  • 토큰 인증. 서버 키 기반 토큰 발급과 WebSocket endpoint 인증 처리.
  • 테스트 자동화. AI 에이전트를 활용해 서버/도구/파이프/유틸 모듈 대상 53개 테스트 케이스 생성.
