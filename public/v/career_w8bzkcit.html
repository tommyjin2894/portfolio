PROJECT 01

AIBOO — 공공기관 현장에 들어갈 음성 비서, 1인 용역으로 직접 개발

Jetson Orin 기반 노인 케어 음성 비서. 비슷한 오픈소스 음성 비서를 시작점으로 받아 단일 Python 서버로 다시 정리하고, 센서·건강·스케줄·긴급·정서지원 같은 도메인 기능을 직접 붙임. 실제 사용 현장에 들어갈 시스템을 1인 용역으로 받아 소프트웨어 전 영역을 혼자 만들고 납품 직전까지 끌고 옴.

프로젝트 배경

공공기관(보훈부) 노인 케어 사업으로 만든 AI 음성 비서. 시니어 사용자가 음성만으로 건강 점검 · 일정 관리 · 긴급 알림 · 정서 지원을 받을 수 있게 한다. 외부 인터넷에 닿지 않는 환경에서 운영해야 하므로 음성 인식·LLM·음성 합성을 모두 기기 안에서 처리하도록 설계. 회사에서 1인 용역으로 배정받아 소프트웨어 전 영역을 혼자 개발 (회로 부분은 외부 지원). 2026.05 납품 예정.

베이스 vs 주요 기여

  • 베이스 — 오픈소스 음성 비서 fork: 활발히 유지되는 음성 비서 오픈소스를 시작점으로 받음. 음성 인식 · LLM · 음성 합성 · 조율 컴포넌트가 Docker 컨테이너 4개로 분리된 구조였고, x86과 Jetson 환경 모두 지원.
  • 주요 기여: Jetson 한 대에서 굴러가도록 컨테이너 4개를 단일 Python 프로세스로 통합하고, Jetson 전용 환경 셋업, 도메인 기능(센서/건강/스케줄/긴급/정서지원/다국어), ESP32 펌웨어 튜닝까지 직접 개발.
  • 브랜치 분리: 공통 음성 비서 코어와 공공기관 도메인 특화 기능(긴급 메시지 자동 전송, 케어센터 연동 등)을 별도 브랜치로 분기 운용.

시스템 구성

주요 기여

  • Docker 4컨테이너 → 단일 Python으로 통합 — 베이스의 음성 인식·LLM·음성 합성·조율 컴포넌트를 한 Python 프로세스로 합치고, 핵심 서비스 3개(메인/도구/홈케어)만 가벼운 프로세스 매니저(PM2)로 분리해 띄움. 홈케어 센서 모듈은 사내 다른 개발자가 먼저 만든 코드를 같은 운영 방식으로 흡수. 메모리 약 750MB 줄어듦 (btop/htop/nvidia-smi 기준).
  • Jetson Orin 환경 구축 — Jetson 전용 OS(JetPack 6) + CUDA 12.6 + Python 3.10 위에서, PyTorch · CTranslate2 · ONNX Runtime을 동작하는 버전 조합으로 직접 묶어 셋업. PyPI 기본 패키지가 Jetson에서 동작 안 해 CTranslate2는 소스에서 직접 빌드, ONNX Runtime은 Jetson 전용 빌드 사용.
  • HTTP + WebSocket 통합 서버 — 하나의 포트로 관리 대시보드, 인증, 펌웨어 업데이트, 이미지 분석, 음성 채팅 연결을 모두 처리.
  • 도메인 기능 구현 — 센서 모니터링 (MQTT 임계값 경보) / 건강관리 (질문 수집 → sentiment 분석 → DB → 월간 달력 뷰) / 스케줄 알림 (daily/weekly/weekdays + 음성 CRUD) / 긴급 감지 (키워드 + 낙상 감지 → emergency 브로드캐스트) / 정서 지원 (시간대별 인사 + idle 대화 + 음악 재생) / i18n (ko/en, 84키).
  • 케어센터 연동 클라이언트·라우팅 엔드포인트 구현 — 케어센터 측이 제공한 API 사양에 맞춰 Jetson 서버 측 연동 클라이언트 구현 (bobo 브랜치): 긴급 메시지 자동 전송, 케어센터 백엔드 연동 hooks, 인증·재시도·에러 처리.
  • VAD/ASR 튜너 페이지 (/tuner) — 환경별 임계값 휴리스틱 튜닝 + DB 저장.
  • Barge-in 구현 — 사용자 발화 감지 시 LLM 토큰 스트리밍 cancel 대신 TTS 재생을 잘라내는 방식 채택. 노인 케어 시나리오에서 동일 UX 달성 가능 + 구현 비용 낮음 → ROI 판단으로 cancel 미구현.
  • SQLite WAL 모드 적용 — 비동기 컴포넌트(스케줄 / MQTT / WebSocket) 동시 쓰기 상황에서 기본 journal 모드 락 충돌이 발생, WAL 모드 전환으로 완화.
  • ESP32 펌웨어 튜닝 — 같은 계열 오픈소스 펌웨어를 베이스로 받아, 빔포밍 칩(XVF3800) 데이터시트가 권장하는 초기화 시퀀스를 그대로 이식해 동작. 마이크 입력과 스피커 출력의 클럭을 분리하는 방식으로 오디오 스트리밍 안정화. 검증은 전체 음성 파이프라인이 정상 작동하는지 기준.

기술 선택

Docker 4컨테이너 → 단일 Python

베이스 오픈소스는 음성 인식·LLM·음성 합성·조율 컴포넌트를 컨테이너 4개로 분리. Jetson 한 대에서 굴리기엔 컨테이너 오버헤드와 GPU 리소스 분할이 부담돼서, 한 Python 프로세스로 합친 뒤 핵심 서비스 3개(메인/도구/홈케어)만 가벼운 프로세스 매니저(PM2)로 분리하는 형태로 변경.

Barge-in: TTS cut 전략

사용자 발화 감지 시 LLM 토큰 스트리밍 cancel 대신 TTS 재생을 잘라내는 방식 채택. 노인 케어 시나리오에서 두 방식 모두 동일 UX 달성 가능 + TTS cut 쪽이 구현 비용 낮음 → ROI 판단으로 cancel 미구현.

마이크 echo 처리 (AEC 미적용)

VAD threshold 동적 조정만으로 TTS 출력 시 self-trigger 제거 확인 (단일 디바이스·1인 발화 환경). AEC(Acoustic Echo Cancellation)는 v1 범위 외로 결정 — 다중 마이크 / 다중 사용자 시나리오에서 v2 검토 과제로 보류.

응답 지연

발화 종료(VAD silence) → STT 결과 반환 체감 약 1.5초 (Faster-Whisper FP16, beam_size 등 디코딩 파라미터는 청취 비교 기반 실측 튜닝). LLM TTFT는 provider별 상이 (Gemini 클라우드 / 로컬 Qwen). 체계적 latency 벤치마크는 v1 범위 외, 납품 후 운영 데이터 기반 v2 과제로 보류.

트러블슈팅 (예시)

PyTorch 최신 버전과 음성 인식 라이브러리의 GPU 라이브러리 충돌

  • 증상: 음성 인식 추론에서 CUDA 에러 발생, 음성 인식이 아예 동작하지 않음
  • 분석: ONNX Runtime을 먼저 의심하다가, PyTorch · CTranslate2 · ONNX Runtime의 버전 조합을 하나씩 바꿔가며 재현 테스트 — PyTorch 최신과 CTranslate2 조합에서만 충돌이 재현됨
  • 해결: PyTorch를 한 단계 이전 버전(2.9.1)으로 다운그레이드 (Jetson 전용 빌드 채널 사용)
  • 결과: 추론 정상 동작. 환경 셋업 가이드에 동작 확인된 버전 조합을 고정해 명시. GPU 라이브러리 버전 차이의 근본 원인 분석은 진행하지 않음 — 동작하는 조합 확보가 v1 우선순위.

한계 및 회고

  • LLM TTFT, 종단 latency 체계적 벤치마크 미실시 — v1 동작 검증 우선, 납품 후 운영 데이터 기반 v2 과제로 보류.
  • 다중 디바이스 동시 호출 시 자원 경합 미검증 — 단일 디바이스 환경(1대 1인 사용자) 가정. 70대 확장 단계에서 클러스터 시나리오 검토 필요.

공통 파이프라인

1. Kneron KL630 (CatchCam — 화재 감지 카메라)

대상 모델: YOLOv5s 6클래스 (Project 04에서 학습)

변환 파이프라인: ONNX → BIE → NEF

Kneron Toolchain (ktc) 양자화 파이프라인 구성: quantize_mode, datapath/weight bitwidth, input/output bitwidth, compiler_tiling, optimize 등 주요 파라미터는 공식 문서 권장 조합 기반으로 세팅.

미지원 연산 처리 (그래프 수술)

미지원 연산(NMS · detect 등 출력 헤드 후처리)은 Netron으로 그래프 시각화 후 onnx Python API로 노드 ID 직접 조회 → 절단·재연결하여 모델 본체만 NPU에 올림. 절단 후 Python 추론으로 모델 본체 동등 동작을 수치 비교 검증한 뒤, 잘려나간 후처리 로직은 SDK 레퍼런스 C 코드를 임베디드 환경에 맞게 통합 (anchor decode · sigmoid · NMS 등은 Kneron/RKNN 제공 레퍼런스 활용).

2. Rockchip RK3588 (AIO3588Q)

  • 대상 모델: YOLO-World (Open-Vocabulary 객체 탐지)
  • 변환 파이프라인: ONNX → RKNN (RKNN Toolkit 기본 INT8 양자화 활용 — 정밀도는 Toolkit 권장 기본값, calibration set은 학습 데이터 일부 사용)
  • AIO3588Q에서 실시간 객체 탐지 추론 검증

Kneron vs Rockchip — 절차 차이

두 NPU 모두 PyTorch → ONNX → 자체 toolchain 변환이라는 공통 패턴이지만, 절차 구성이 달랐다. RKNN은 toolkit이 비교적 정리되어 있어 변환 자체는 짧게 끝났고, Kneron은 BIE → NEF로 이어지는 단계가 많아 변환 실패 시 공식 문서 기반으로 설정을 반복 조정. 미지원 연산은 모델 그래프에서 제거하고 CPU 측 후처리로 분리.

트러블슈팅 (예시) — KL630 미지원 연산 처리

  • 증상: YOLOv5s를 BIE로 변환하는 과정에서 출력 헤드의 후처리 영역(NMS 등)에 KL630 미지원 연산이 포함되어 변환 실패
  • 분석: ONNX 그래프 상 출력 헤드의 후처리 부분(detect 레이어, NMS)이 KL630 NPU에서 미지원 연산으로 구성되어 있음을 확인
  • 해결: Netron으로 그래프 시각화 → 절단 위치 노드 ID 직접 조회 → onnx Python API로 노드 제거·재연결 (모델 본체만 NPU 변환) → Python 추론으로 모델 본체 동등 동작 수치 비교 검증 → 잘려나간 후처리 로직은 SDK 레퍼런스 C 코드를 임베디드 환경에 맞게 통합 (anchor decode·sigmoid·NMS는 SDK 제공 레퍼런스 활용)
  • 결과: NPU 추론 + CPU 후처리 조합으로 정상 동작, 변환된 모델 크기/성능 모두 만족

환경

Docker 기반 KL630 전용 크로스 컴파일 환경 — 재현 가능하게 구성.

한계 및 회고

  • 양자화 전후 mAP 정량 비교는 v1 범위 외 — W&B 등 실험 추적 도구 도입과 함께 v2 과제로 보류.
  • C 후처리 코드는 SDK 레퍼런스 통합 수준 — anchor decode·dequantization 직접 작성 경험은 추가 학습 영역.

PROJECT 05

LLM 풀 파이프라인 직접 자동화 (학습 → 변환 → 양자화 → 서빙)

AIBOO 음성 비서의 LLM 모듈로 Qwen 모델을 직접 파인튜닝. 외부 API를 못 쓰는 환경이라, 더 큰 LLM이 만든 응답을 모아 학습 데이터셋을 직접 구축 (모델 내부 확률값 대신 출력 텍스트만 사용하는 방식). 데이터 생성 → 학습 → 모델 합치기 → 압축 → 로컬 서빙 등록까지 자체 스크립트로 한 번에 돌아가게 자동화.

기간
2025.12 ~ 현재
역할
합성 데이터 + LoRA 학습 + GGUF 양자화 + Ollama 서빙
연계
Project 01 (AIBOO LLM 모듈)
카테고리
LLM
스택
Qwen · PEFT · TRL · llama.cpp · Ollama

전체 파이프라인

1. 합성 데이터 생성 파이프라인

  • 방식: 상위 모델(gpt-oss / Qwen 상위급)을 ollama로 로컬 구동, 다양한 프롬프트로 응답 샘플링해 합성 학습 데이터 직접 구축. logits distillation이 아니라 출력 텍스트 기반 SFT 방식 (Self-Instruct 계열에 가까움)
  • 외부 API 미사용 — 망분리·정책 제약으로 상용 LLM API 사용 불가, 모든 데이터셋을 로컬 추론으로 생성
  • 다양성 확보 장치: 15개 상황 카테고리 + 8개 감정 톤 배치마다 랜덤 조합 / 중복 instruction 자동 필터링 / 길이·표현·이모지 다양성 강제 프롬프트
  • STT 스타일 강제: 구두점 최소, 구어체, 간투사("어/음/그/아"), 띄어쓰기 불완전 등 — 음성 입력에 최적화

2. 페르소나 학습 — "아이부"

  • 200% 과몰입 공감 캐릭터 (감정 우선, 반말, 이모지 적극)
  • 도메인 적용(노인 케어)은 시스템 프롬프트 단계에서 처리 — 페르소나는 학습으로, 도메인은 프롬프트로 분리
  • 학습 데이터 example: {"instruction": "어 오늘 회사에서 완전 깨졌어", "output": "아 진짜? ㅠㅠ 고생한 너 생각하니까 내가 다 속상하고 화난다 진짜!"}

3. 모델 학습 (src/train_qwen.py) — 4크기 비교 학습

  • 4크기 비교 학습 이유: 너무 작은 모델은 추론 시 엉뚱한 응답이 자주 나오고, 너무 큰 모델은 Jetson 추론 비용 부담 — 응답 품질 vs 모델 크기 스윗 스팟 탐색을 위해 직접 비교 학습 후 선택하는 방식 채택
  • 베이스: Qwen2.5-0.5B-Instruct (master), 브랜치별 다른 크기 학습 (Qwen 1.5B / Qwen3 1.7B / Qwen3 4B Q4_K_M)
  • 방식: 표준 LoRA + SFTTrainer (TRL 라이브러리)
  • LoRA 설정: r=8, lora_alpha=16, lora_dropout=0.1, target_modules: q_proj, k_proj, v_proj, o_proj, gate_proj
  • 학습 설정: batch=1 + gradient_accumulation=4 (effective batch=4), lr=2e-4, weight_decay=0.001, fp16=True, max_grad_norm=0.3, lr_scheduler=constant, warmup_ratio=0.03, optim=adamw_torch
  • 포맷: ChatML (<|im_start|> / <|im_end|>)
  • 모니터링: TensorBoard

4. 양자화 및 로컬 서빙

학습된 LoRA 어댑터를 베이스 모델에 합치고, GGUF 포맷으로 변환 후 Q4_K_M으로 양자화한 뒤 로컬 LLM 서버(Ollama)에 등록하는 흐름을 한 번에 돌아가게 자체 스크립트로 묶음. 모델 메타파일에 응답 종료 토큰을 명시해 추론 시 토큰 누수 방지.

평가

  • 응답 품질 정성 평가 (시간 제약으로 perplexity 등 정량 metric 미실시)
  • 9:1 train/test 분리

향후 계획

  • Multi-LoRA: 질의 난이도 분류 어댑터를 별도 학습하여 응답 어댑터와 동적 라우팅 (현재는 단일 어댑터)
  • 정량 평가 도입 (perplexity, 자체 eval set의 LLM-as-judge)

한계 및 회고

  • 모델 4크기 간 정량 비교(perplexity / LLM-as-judge)는 v2 과제 — 현재는 정성 평가 기반 스윗 스팟 결정.
PROJECT 06

MCP 도구 릴레이 서버 (FastMCP 기반)

FastMCP 도구를 WebSocket으로 외부에 서빙하는 3계층 릴레이. AIBOO 시스템의 도구 통합 모듈로 사용.

기간
2026.03 ~ 현재
역할
구조 설계 + 자체 MCP 도구 구현 + 테스트 자동화
연계
Project 01 (AIBOO 시스템 일부)
카테고리
백엔드
스택
FastAPI · FastMCP · WebSocket · Gemini · Kakao Local

구성

주요 기여

  • 3계층 WebSocket 릴레이 구성 — 소비자 ↔ 서버 ↔ 파이프 ↔ stdio 도구
  • 요청 ID 매핑 + 양방향 메시지 라우팅 — 여러 소비자가 동시에 보낸 요청 ID를 릴레이 내부 ID로 바꿔 도구에 전달, 응답 돌아올 때 원래 ID로 복구
  • stdio ↔ WebSocket 자동 재연결 로직 — exponential backoff 기반 (1초 → 최대 30초 cap), max_retries 이후 상위 핸들러에 에러 전파
  • 토큰 기반 인증 — 서버 키 자동 생성, agentId=<name> 형태 토큰 발급
  • 자체 MCP 도구 구현 — calculator (Python math/random 수식), recommend_what_to_do (Gemini 2.5 Flash + Kakao Local API로 위치 기반 음식·숙박·관광·활동 추천)
  • 테스트 자동화 — Claude CLI agent 병렬 실행으로 53개 테스트 케이스 자동 생성 (서버/도구/파이프/유틸 모듈 대상)

API 엔드포인트

  • GET /mcp_endpoint/ — 서버 상태 · GET /mcp_endpoint/health?key=<key> — 헬스 체크
  • WS /mcp_endpoint/mcp/?token=<token> — 파이프 연결 · WS /mcp_endpoint/call/?token=<token> — 소비자 연결

한계 및 회고

  • 자동 생성 테스트의 정량 커버리지는 빠르게 확보되지만, 핵심 시나리오의 의도성은 별도 검증 필요
