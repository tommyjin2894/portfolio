주요 프로젝트

01 AIBOO — 공공기관 현장에 들어갈 음성 비서, 1인 용역으로 직접 개발 2025.02 ~ 현재 · 1인 용역 · 보훈부 노인 케어
실제 사용 현장에 들어갈 시스템을 1인 용역으로 받아 소프트웨어 전 영역 혼자 만들고 납품 직전까지 끌고 옴 (회로 부분은 외부 지원), 2026.05 납품 예정. 비슷한 오픈소스 음성 비서(컨테이너 4개 분리 구조)를 시작점으로 받아 단일 Python 서버로 통합 — 메모리 약 750MB 절감. 다른 개발자가 만든 홈케어 센서 모듈을 같은 운영 방식으로 흡수해 핵심 서비스 3개로 분리. DB는 비동기 컴포넌트들이 동시 쓰기에 락 충돌이 생겨 WAL 모드로 정리. Jetson 환경에서 음성 인식 라이브러리 소스 빌드 + GPU 라이브러리 버전 호환성 디버깅. 외부 케어센터의 API 사양에 맞춰 연동 클라이언트와 라우팅 엔드포인트를 직접 구현 (긴급 메시지 자동 전송 등).
스택: Jetson Orin · CTranslate2 · aiohttp · Faster-Whisper · Qwen · MQTT · PM2 · SQLite WAL · ESP-IDF
02 LLM 풀 파이프라인 자체 자동화 (학습 → 변환 → 양자화 → 서빙) 2025.12 ~ 현재 · AIBOO LLM 모듈
외부 API를 못 쓰는 환경이라 더 큰 LLM(gpt-oss · Qwen 상위급)이 만든 응답을 모아 학습 데이터셋을 직접 구축 (모델 내부 확률값 대신 출력 텍스트만 사용). Qwen 4가지 크기(0.5B / 1.5B / 3-1.7B / 3-4B)를 LoRA로 비교 학습해 응답 품질 vs 크기 스윗 스팟 탐색. 데이터 생성 → 학습 → 모델 합치기 → 압축 → 로컬 서빙 등록까지 자체 스크립트로 한 번에 돌아가게 자동화.
스택: Qwen · PEFT · TRL · llama.cpp · GGUF · Ollama · 자체 자동화
03 MCP 도구 릴레이 서버 — 3계층 WebSocket + stdio 자동 재연결 2026.03 ~ 현재 · AIBOO 도구 통합
기존 챗봇 서버 코드를 건드리지 않고 외부 도구 호출 영역만 분리하기 위한 별도 서버. 클라이언트 ↔ 서버 ↔ 파이프 ↔ 도구 프로세스의 3계층 릴레이 구조 직접 설계 — 요청 ID 변환, 토큰 기반 인증, 끊긴 연결 자동 재시도(점진 backoff), 자체 도구(계산기·위치 기반 추천) 구현.
스택: FastAPI · WebSocket · FastMCP · Gemini · Kakao Local API
04 Multi-NPU 모델 포팅 + ONNX 그래프 수술 2025.02 ~ 09
서로 다른 NPU 두 종(Kneron KL630 · Rockchip RK3588)에 YOLO 객체 탐지 모델을 직접 올림 — 양쪽 다 추론 검증 완료. 출력 헤드의 후처리(NMS · detect 등)가 NPU에서 지원되지 않아, 모델 그래프를 시각화 도구로 보고 해당 노드를 직접 잘라낸 뒤 다시 연결해 모델 본체만 NPU에 올리고, 잘려나간 후처리는 SDK 레퍼런스 코드를 임베디드 환경에 통합. 모델 변환·양자화·검증 파이프라인을 환경별로 직접 구성.
스택: Kneron Toolchain · RKNN Toolkit · ONNX · Netron · ONNX Python API
05 Telemedicine 백엔드 (1:N 화상 진료 PoC) + Piper TTS 한국어 G2P 2025
Telemedicine (2025년 약 3개월, PoC·시연 단계 완료) — WebRTC 기반 의사 1인 ↔ 환자 다수 화상 진료 백엔드를 직접 개발 + NAT 통과용 미디어 릴레이 서버(Coturn) 직접 셋업 + 시그널링 서버 구현. 본인 코드를 후속 개발자가 다른 언어로 인계받아 리팩토링 진행 중. Piper TTS 한국어 G2P 토크나이저 직접 통합 — 기본 음소화 도구가 한국어 처리에 약해, 학습/추론 양쪽 파이프라인에 한국어 G2P를 직접 결합 (공식 가이드가 없는 영역).
스택: FastAPI · WebRTC · Coturn · WebSocket · PyTorch Lightning · 한국어 G2P
