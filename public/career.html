프로젝트 배경

공공기관(보훈부)의 노인 케어 사업 일환으로, 시니어 사용자가 음성 대화만으로 건강 상태 점검 · 일정 관리 · 긴급 상황 알림 · 정서 지원 등을 받을 수 있는 AI 음성 비서. 개인정보 보호와 망 분리 환경 운영이 요구되어 모든 추론(ASR/LLM/TTS)을 온디바이스에서 수행하는 것을 설계 전제로 함. 단독 개발자로 기획부터 구현·운영·납품 준비까지 전 과정을 담당.

베이스 vs 본인 기여

  • 베이스 — xiaozhi-esp32-server 계열 fork (m5chatbot-server): x86 / Jetson 양 환경을 지원하는 Docker 4컨테이너 구성으로 ASR · LLM · TTS · orchestrator를 분리. 중국 커뮤니티에서 활발히 유지되는 음성 비서 오픈소스를 시작점으로 채택.
  • 본인 기여 — aiboo-server-lite: Jetson Orin 단일 디바이스 환경에 맞춰 컨테이너를 단일 Python 프로세스로 통합 리팩토링 + Jetson 전용 환경 셋업 + 도메인 기능 8종(센서·건강·스케줄·긴급·정서지원·i18n 등) 자체 구현 + ESP32 펌웨어 튜닝(빔포밍·암호화) 까지 단독 개발.
  • 브랜치 분리 — main / bobo: main은 일반 음성 비서 코어, bobo는 공공기관 도메인 특화 hooks(긴급 메시지 자동 전송, 케어센터 연동 등) 분리하여 코어 재사용성 확보.

시스템 구성

주요 기여

  • Docker 4컨테이너 → 단일 Python 통합 리팩토링 — m5chatbot-server의 ASR/LLM/TTS/orchestrator 분리 컨테이너 구성을 단일 aiohttp 프로세스로 통합. PM2로 3개 핵심 서비스(aiboo/mcp/homecare)만 분리하여 장애 격리는 유지. 메모리 사용량 약 750MB 절감 (btop/htop/nvidia-smi 기준).
  • Jetson Orin 환경 구축 — JetPack 6 · CUDA 12.6 · Python 3.10 환경에서 PyTorch 2.9.1 (cuBLAS 충돌 회피 위해 2.11+ 사용 금지) + CTranslate2 4.7.1 CUDA 소스 빌드 + onnxruntime-gpu Jetson 전용 빌드 통합. PyPI 기본 패키지 미동작 이슈 해결.
  • HTTP + WebSocket 통합 서버 — 단일 포트 8003에서 대시보드 / Admin / OTA / 비전 분석(/mcp/vision/explain) / WebSocket(/xiaozhi/v1/) 모두 처리.
  • 도메인 기능 8종 자체 구현 — 센서 모니터링 6종 (MQTT 임계값 경보) / 건강관리 (질문 수집 → sentiment 분석 → DB → 월간 달력 뷰) / 스케줄 알림 (daily/weekly/weekdays + 음성 CRUD) / 긴급 감지 (키워드 + 낙상 감지 → emergency 브로드캐스트) / 정서 지원 (시간대별 인사 + idle 대화 + 음악 재생) / i18n 84키 (ko/en).
  • VAD/ASR 튜너 페이지 (/tuner) — 환경별 임계값 휴리스틱 튜닝 + DB 저장.
  • Barge-in 구현 — LLM 토큰 스트리밍 cancel 대신 TTS 재생을 잘라내는 방식. LLM streaming cancel은 model state 관리 복잡도가 높아 회피 → 자연스러운 대화 흐름 + 구현 단순성 확보.
  • SQLite WAL 모드 동시성 처리 — 비동기 컴포넌트 (스케줄 / MQTT / WebSocket)가 동시 쓰기로 발생하던 락 충돌을 WAL 모드 전환으로 해결.
  • ESP32 펌웨어 튜닝 — m5chatbot-build 베이스에서 빔포밍 (XVF3800 I2C 제어) 적용, I2S Split (RX Slave + TX Master) 오디오 스트리밍, AES-128-CTR 암호화 통합.

기술적 판단

Docker 4컨테이너 → 단일 Python

기존 m5chatbot-server는 ASR/LLM/TTS/orchestrator를 컨테이너 분리. Jetson 환경에선 컨테이너 오버헤드와 GPU 리소스 분할이 비효율적이라 판단, 단일 프로세스 통합 + PM2 3서비스로 장애 격리는 유지하는 절충안 선택.

Barge-in: TTS cut 전략

사용자 발화 감지 시 LLM 토큰 스트리밍 cancel은 model state 관리 복잡도가 높음. LLM은 그대로 두고 TTS 재생만 잘라내는 방식 채택 → 자연스러운 대화 흐름 확보 + 구현 단순화.

마이크 echo 처리 (AEC 미적용)

VAD threshold 동적 조정만으로 TTS 출력 시 self-trigger 회피 가능. AEC(Acoustic Echo Cancellation)는 단일 디바이스 환경에서 알고리즘 복잡도 대비 이득이 작다고 판단.

응답 지연

발화 종료(VAD silence) → STT 결과 반환 약 1.5초 (Faster-Whisper FP16 기준). LLM TTFT는 provider별 상이 (Gemini 클라우드 / 로컬 Qwen). 정밀 측정 및 최적화는 운영 안정화 우선으로 후순위.

트러블슈팅 (예시)

PyTorch 2.11 + CTranslate2 cuBLAS 충돌

  • 증상: Faster-Whisper 추론에서 CUDA 에러 발생, STT 동작 불가
  • 분석: onnxruntime 의심 → 격리 테스트로 PyTorch 단독 cuBLAS 호출 시 충돌 확인 → CTranslate2와의 cuBLAS 버전 호환성 이슈로 좁힘
  • 해결: PyTorch 2.11 → 2.9.1 다운그레이드 (Jetson 전용 wheel 채널)
  • 결과: 추론 정상 동작, 환경 셋업 가이드에 "PyTorch 2.11+ 사용 금지" 박제

한계 및 회고

  • LLM TTFT, 종단 지연 정밀 측정 미실시 (안정화 우선)
  • 다중 디바이스 동시 호출 시 자원 경합 미검증 (단일 디바이스 환경 가정)

공통 파이프라인

1. Kneron KL630 (CatchCam — 화재 감지 카메라)

대상 모델: YOLOv5s 6클래스 (Project 04에서 학습)

변환 파이프라인: ONNX → BIE → NEF

Kneron Toolchain (ktc) 양자화 파라미터 튜닝: quantize_mode (default / post_sigmoid), datapath/weight bitwidth (int8/16/int4 mix), input/output bitwidth, compiler_tiling, optimize 등 8개 주요 파라미터를 모델 특성에 맞춰 조합.

ONNX 그래프 편집

KL630 미지원 op (NMS 등 후처리 영역)는 onnx 툴킷으로 노드 제거 후, 제거된 후처리 연산을 호스트 측 C 애플리케이션에서 재구현하여 정확도 유지.

2. Rockchip RK3588 (AIO3588Q)

  • 대상 모델: YOLO-World (Open-Vocabulary 객체 탐지)
  • 변환 파이프라인: ONNX → RKNN, INT8 양자화로 모델 크기 축소
  • AIO3588Q에서 실시간 객체 탐지 추론 검증

Kneron vs Rockchip — 난이도 비교

두 NPU 모두 PyTorch → ONNX → 자체 toolchain 변환이라는 공통 패턴을 따르지만, 실제 작업 난이도는 크게 달랐습니다. Kneron KL630이 훨씬 까다로웠습니다. RKNN은 toolkit이 비교적 잘 정리되어 있고 미지원 op도 적어 변환 자체는 1~2일이면 충분했지만, Kneron은 BIE → NEF로 이어지는 변환 단계가 많고, 양자화 비트폭을 노드별로 세밀히 조정해야 했으며, 미지원 op는 호스트 CPU 측에서 별도 후처리로 옮겨야 하는 복잡성이 있었습니다.

트러블슈팅 (예시) — KL630 미지원 op 처리

  • 증상: YOLOv5s를 BIE 변환할 때 NMS 등 후처리 영역의 특정 op가 KL630에서 미지원으로 fail
  • 분석: ONNX 그래프를 살펴보니 출력 헤드의 후처리 부분(detect 레이어, NMS)이 KL630 NPU에서 실행 불가능
  • 해결: onnx 툴킷으로 해당 노드들을 그래프에서 제거 (모델 본체만 NPU 변환) + 잘려나간 후처리 연산을 호스트 측 C 애플리케이션에서 직접 재구현하여 정확도 보존
  • 결과: NPU 추론 + CPU 후처리 조합으로 정상 동작, 변환된 모델 크기/성능 모두 만족

환경

Docker 기반 KL630 전용 크로스 컴파일 환경 — 재현성 확보, 팀 간 일관성.

한계 및 회고

  • 양자화 전후 mAP 손실 정량 비교 데이터 미보존 (이후 W&B 등 실험 추적 도구 필요성 인지)
  • KL630 calibration dataset 구성에 대한 체계적 비교 미실시

  • 방식: 로컬 오픈소스 LLM(ollama 기반) 응답을 distillation해서 학습 데이터 생성
  • 외부 API 미사용 — 학습 데이터셋 구축에 외부 LLM API를 쓰지 않음 (정책/비용 회피)
  • 다양성 확보 장치: 15개 상황 카테고리 + 8개 감정 톤 배치마다 랜덤 조합 / 중복 instruction 자동 필터링 / 길이·표현·이모지 다양성 강제 프롬프트
  • STT 스타일 강제: 구두점 최소, 구어체, 간투사("어/음/그/아"), 띄어쓰기 불완전 등 — 음성 입력에 최적화

2. 페르소나 학습 — "아이부"

  • 200% 과몰입 공감 캐릭터 (감정 우선, 반말, 이모지 적극)
  • 도메인 적용(노인 케어)은 시스템 프롬프트 단계에서 처리 — 페르소나는 학습으로, 도메인은 프롬프트로 분리하여 재사용성 확보
  • 학습 데이터 example: {"instruction": "어 오늘 회사에서 완전 깨졌어", "output": "아 진짜? ㅠㅠ 고생한 너 생각하니까 내가 다 속상하고 화난다 진짜!"}

3. 모델 학습 (src/train_qwen.py)

  • 베이스: Qwen2.5-0.5B-Instruct (master), 브랜치별 다른 크기 학습 (Qwen 1.5B / Qwen3 1.7B / Qwen3 4B Q4_K_M)
  • 방식: 표준 LoRA + SFTTrainer (TRL 라이브러리)
  • LoRA 설정: r=8, lora_alpha=16, lora_dropout=0.1, target_modules: q_proj, k_proj, v_proj, o_proj, gate_proj
  • 학습 설정: batch=1 + gradient_accumulation=4 (effective batch=4), lr=2e-4, weight_decay=0.001, fp16=True, max_grad_norm=0.3, lr_scheduler=constant, warmup_ratio=0.03, optim=adamw_torch
  • 포맷: ChatML (<|im_start|> / <|im_end|>)
  • 모니터링: TensorBoard

4. 양자화 및 로컬 서빙

# HF → GGUF
python llama.cpp/convert_hf_to_gguf.py outputs/merged \
    --outfile outputs/qwen.gguf

# Q4_K_M 양자화
llama.cpp/build/bin/llama-quantize \
    outputs/qwen.gguf outputs/qwen_q4km.gguf q4_k_m

# Ollama 등록
ollama create aiboo_q4km -f Modelfile

Modelfile에 ChatML stop tokens (<|im_start|>, <|im_end|>) 명시 — 추론 시 토큰 누수 방지.
